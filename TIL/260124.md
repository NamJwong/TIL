## 코드 저장할 때의 auto lint fix 고장 해결

- 원인
  1. ESLint 서버가 죽었던 것
  2. settings.json의 다음 설정이 누락돼있었음. (그런데 기존에는 왜 됐는지는 못 밝혀냄.)
  ```json
  {
    "editor.codeActionsOnSave": {
      "source.fixAll.eslint": "explicit"
    }
  }
  ```
- 느낀 점
  - 복잡한 원인도 아닌데 헤맨 게 부끄럽다. 그래서 왜 지금까지 몰랐는지 돌아봤다.
  - 디버깅에서 집요함 부족: 지난번에 1번과 같은 문제가 있었을 때 ESLint 서버가 문제인가 하고 로그는 확인해봤는데, 로그에 error라는 단어가 없어서 넘어갔다. 하지만, 정상 상태일 때의 로그가 뭔지 모르니 짚어봐야 했다. AI한테 물어보면 바로 알려주는데.
  - 무지성 디버깅: 옛날도 같은 문제를 종종 접하고 구글링을 통해 2번처럼 해결했는데 하라는대로 하고 해결이 되면 왜 해결이 된 건지를 돌아보지 않았다. 그래서 이번에도 settings.json의 저 곳을 봐야겠다는 생각을 하지 못했다. 이런 것도 이젠 AI한테 물어보면 바로 알려주니까 간단히라도 알고 넘어가자.

## 이것저것 해볼만한 것 아이디어

- 정확히는 '건영이의' 아이디어. 모각작 하다가 나처럼 일을 하지 않는 상황에 성장을 위해 어떤 걸 해볼 수 있을지 박교수님께 자문해봤다.
- 아이디어 목록
  1. 해결하고 싶은 문제 정의 -> 사이드 프로젝트
  - 되든 안되든 사업한다는 마인드로 문제 정의부터
  2. AI 에이전트 만들기
  - 에이전트 프레임워크 활용해봐도 좋다. ex 마스트라, Tanstack AI, 코파일럿킷
  - 개발 워크플로우에서 사람이 해야만 했던 '모호한 일'을 시켜보기
  - 에이전트 서비스 만드는 언어의 주류가 타입스크립트인 것은 찬스다
  3. 전 직장에서 만났지만 해결하지 못한 문제를 쓸만하게 재현해보기
  4. ESLint 플러그인
  5. Tanstack query 같은 라이브러리 뜯어보고 비슷하게 구현해보기
  - 잘 쓰고 싶다는 니즈가 있는데, 잘 이해하면 잘 쓰게 된다.
  - 좋은 인터페이스가 많아서 많이 배운다.
- 그외 인사이트
  - 기술이든 제품이든 '\_\_\_는 무슨 문제를 풀고 싶었을까?'의 관점으로 생각해보는 것이 중요하다. 내가 보는 모든 것에 대해 '이것은 무슨 문제를 해결하는가' 생각해보기 ex npm, dom, Cursor...
  - 개발자는 추상화된 현상(ex 사이트가 너무 느려요)을 진짜 문제로 정의할 수 있어야 한다. 그리고 좋은 해결 방법을 도출하기 위한 여러 안을 알고 있어야 한다.(하드 스킬) 해결 방법을 선정하기 위한 현재 상황의 조건(ex 리소스, 영향 범위)를 파악할 수 있어야 한다. => 이를 종합하면 솔루션에 대한 근거가 된다.
